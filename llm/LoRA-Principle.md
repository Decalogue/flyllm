# LoRA 的原理是什么？rank 和 alpha 参数怎么选？为什么有效？

## 1. 核心定性
本质上，**LoRA (Low-Rank Adaptation)** 是一种基于低秩矩阵分解的参数高效微调方法，通过在预训练权重旁路注入可训练的低秩矩阵 $BA$（其中 $B \in \mathbb{R}^{d \times r}, A \in \mathbb{R}^{r \times k}$，$r \ll \min(d,k)$），在冻结原权重的基础上，仅优化极少量参数（0.1%-1%）即可达到全量微调效果。

## 2. 具体流程
1. **权重分解**：对预训练权重 $W_0 \in \mathbb{R}^{d \times k}$，LoRA 注入更新 $\Delta W = BA$，其中 $B$ 随机初始化（高斯），$A$ 全零初始化，确保训练起始时 $\Delta W = 0$
2. **前向传播**：$h = W_0 x + \Delta W x = W_0 x + B A x$，冻结 $W_0$，只训练 $B$ 和 $A$
3. **参数选择**：rank $r$ 控制低秩程度（通常 4-64），alpha $\alpha$ 控制缩放因子 $\alpha / r$，用于平衡预训练权重和适配器权重的贡献

## 3. 数学基础
**权重更新公式**:
$$W' = W_0 + \Delta W = W_0 + BA$$

**前向计算**:
$$h = (W_0 + BA)x = W_0 x + B(Ax)$$

**梯度计算**:
$$\frac{\partial \mathcal{L}}{\partial A} = \frac{\partial \mathcal{L}}{\partial h} \cdot B$$
$$\frac{\partial \mathcal{L}}{\partial B} = \frac{\partial \mathcal{L}}{\partial h} \cdot A^T$$

**参数数量对比**:
- 全量微调参数: $d \times k$
- LoRA 参数: $r(d + k)$
- 压缩比: $\frac{r(d+k)}{dk} \approx \frac{2r}{\min(d,k)}$

**实践参数**:
- BERT-base: $d=768, k=768$, $r=16$ → 压缩比 4.2%
- LLaMA 7B: $d=4096, k=4096$, $r=16$ → 压缩比 0.78%

**rank 选择理论**:
权重更新本征维度 $r_{intrinsic}$ 通常很小。对于 NLP 任务，$r_{intrinsic} \approx 10-50$，远小于权重维度。LoRA 用低秩矩阵捕捉此本质更新。

**alpha 的作用**:
$$W' = W_0 + \frac{\alpha}{r} BA$$

- $\alpha$ 控制适配器权重相对于 $W_0$ 的贡献
- 实践中 $\alpha$ 通常设为 $2r$（如 $r=16, \alpha=32$）
- 等效学习率：$\alpha$ 越大，BA 更新越大，类似于学习率放大

**复杂度分析**:
- **训练**：额外计算 $BAx$ 增加 10-20% 计算量，但反向传播仅需计算 $B,A$ 梯度，整体速度提升 1.5-2 倍
- **推理**：合并权重 $W' = W_0 + BA$，零推理开销，速度不变

## 4. 工程考量
**为什么 LoRA 有效**:

1. **本征维度假设**：预训练权重占据低维流形，任务适配只需在低维子空间调整。实验表明，即使 rank=1，LoRA 在多数任务上达到 90% 全量微调效果

2. **参数隔离**：BA 学习任务特定知识，$W_0$ 保留通用知识。切换任务只换 BA（仅占 1% 参数），实现参数高效多任务

3. **初始化敏感性**：全零初始化 $A$ 保证训练从 $W_0$ 开始，避免破坏预训练表示。随机初始化 $B$ 提供足够多样性

**rank 选择策略**:

| rank | 参数占比 | 适用任务 | 效果 |
|------|---------|----------|------|
| 1-4 | 0.1-0.4% | 简单任务（二分类） | 85-90% 全量效果 |
| 8-16 | 0.5-1% | 标准 NLP（NER/QA） | 95-98% 全量效果 |
| 32-64 | 2-4% | 复杂任务（生成） | 99%+ 全量效果 |
- **经验法则**: $r=16$ 是 BERT/LLaMA 的通用选择，覆盖 95% 任务
- **自动选择**: AdapterHub 提出的 AutoLoRA 根据梯度矩阵本征值自动选 $r$，在 GLUE 上平均比固定 $r=16$ 提升 0.8 点

**alpha 调优**:
- **alpha = r**: BA 权重与 $W_0$ 同等重要，适合需要大幅调整的任务
- **alpha = 2r**: BA 权重是 $W_0$ 两倍，实践中推荐，平衡稳定性和效果
- **alpha = 0.5r**: 保守更新，适合微调大模型防止灾难性遗忘

**Trade-off**:
- **Rank vs 效果**: $r>64$ 后收益递减，GLUE 上 $r=64$ 比 $r=16$ 仅提升 0.3 点，但参数增加 4 倍
- **Rank vs 内存**: $r=32$ 时 LLaMA 7B 可训练总参数 4.2M（显存 8GB），$r=64$ 时 8.4M（显存 10GB）

**致命弱点**:
- **秩瓶颈**: 对于需要全秩更新的任务（如多任务学习），低秩假设失效，性能比全量微调低 5-10%
- **层选择敏感**: 只应用于 $W_q,W_v$ 效果最佳，应用于所有层反而下降。错误层选择导致效果波动 3-5 点
- **灾难性遗忘**: 尽管只更新 1% 参数，过度训练仍导致通用能力下降，$r$ 越大遗忘越严重

## 5. 工业映射
在工业界，LoRA 是 **大模型微调的事实标准**：
- **HuggingFace PEFT**: `peft.LoraConfig` 提供统一接口，rank=16, alpha=32 成为默认配置。在 GPT-3 175B 微调中，仅训练 0.08% 参数（140M），在 CoNLL-2003 上 F1 92.1（vs 全量 92.3），训练时间从 3 周降至 3 天
- **LLaMA-Adapter**: LLaMA 2 的官方 LoRA 实现，rank=16, alpha=32，7B 模型在 Alpaca 数据上微调后 MT-Bench 得分 6.8。关键优化：只应用于 q_proj 和 v_proj，跳过 k_proj 和 o_proj 以减少 50% 参数
- **Microsoft LoRA Hub**: 在 Azure ML 上，LoRA 适配器作为独立 artifact 存储。基础模型 LLaMA-65B 共享，每个任务只存储 100MB 适配器（vs 120GB 完整模型），存储成本降低 1200 倍
- **多租户服务**：在推理时，加载基础模型一次，为每个用户动态加载其 LoRA 适配器，显存仅增加 1-2%，支持 1000+ 用户共享同一模型，推理成本降低 90%
- **量化 + LoRA**：QLoRA 将 LoRA 与 4-bit 量化结合，rank=64, alpha=16，在 48GB GPU 上微调 65B 模型，保持 16-bit 更新精度，效果与 16-bit 全量相当。这是当前边缘部署的主流方案

**最佳实践**：
- **Rank 选择**: 从 r=16 开始，若效果不足 → r=32 → r=64，每次倍增
- **Alpha 选择**: 固定 alpha=2r，无需调优
- **层选择**: 只应用于 q_proj 和 v_proj，跳过 k_proj, o_proj
- **学习率**: 比全量微调大 5-10 倍（如 lr=1e-3 vs 1e-4），因为 BA 权重需要从 0 快速学习

**趋势**：QLoRA + LoRA 成为大模型微调的金标准，在消费级 GPU（24GB）上可微调 70B 模型，民主化大模型微调
