# 词汇表大小如何确定？太大会怎样，太小会怎样？

## 1. 核心定性
本质上，**词汇表大小**是 tokenizer 空间复杂度与序列长度时间复杂度的核心权衡参数，通过信息论中的熵率-词表大小权衡曲线在模型容量、推理效率和泛化能力间寻求帕累托最优。

## 2. 具体流程
1. **目标函数**：在训练语料 $C$ 上最小化总计算成本：$\text{Cost}(|V|) = \lambda_1 \cdot |V| \cdot d_{model} + \lambda_2 \cdot \text{avg_len}(tokenize(C))$，其中第一项为嵌入矩阵参数量，第二项为序列长度相关的注意力计算成本
2. **分词效率估算**：使用 BPE/UniGram 算法在不同词表大小 $|V| \in [10k, 100k]$ 上训练临时词表，绘制词压缩率曲线，选择拐点后收益递增变缓的词表大小（通常 30-50k）
3. **下游验证**：在 $|V|$ 候选集合 ${30k, 50k, 100k}$ 上训练小规模模型（1B 参数量），监控词表外率（OOV 率）和 <unk> token 频率，确保 OOV < 0.1%

## 3. 数学基础
设语料 $C = \{x_1, x_2, ..., x_N\}$，词表为 $V$，分词函数 $T: \Sigma^* \to V^*$。

**词表复杂度**：
$$|V| = |V_{\text{base}}| + k$$
其中 $V_{\text{base}}$ 是基础 token，$k$ 是优化变量。

**平均序列长度**：
$$L(k) = \frac{1}{N}\sum_{i=1}^N |T_k(x_i)|$$

**总参数量**：
$$P(k) = (|V_{\text{base}}| + k) \times d$$

**计算成本模型**：
$$\text{Cost}(k) = \alpha \cdot P(k) + \beta \cdot (L(k))^2 \cdot d \cdot \text{layers}$$

**最优解**：
$$k^* = \arg\min_{k \in [10k, 200k]} \text{Cost}(k)$$

**信息熵下界**：
$$H(C) \leq \log_2 |V|$$
$$|V|_{\min} = 2^{H(C)} \approx 2^{12} = 4096 \text{ (理想情况)}$$

**实践公式**：
$$|V|_{\text{optimal}} \propto \sqrt{N} \text{ where } N \text{ 为训练 token 数}$$
典型：$N=1T \Rightarrow |V| \approx 32k$ to $64k$

**过拟合度量**：
对于低频 token $v$：
$$\text{freq}(v) < \frac{10}{d} \Rightarrow \text{under-fitting}$$

## 4. 工程考量
**太小的影响**:
-   **序列长度爆炸**  ：$|V| = 10k$ 时，`"internationalization"` 需拆为 17 个 token（`"in" + "ter" + "..."`），导致 $L$ 增加 3-4 倍，推理延迟增加 50%
-   **语义破碎**  ：专业术语（"transformer", "attention"）被过度切分，embedding 失去专有概念表示，下游任务准确率下降 3-5 点
-   **OOV 率飙升**  ：真实场景 OOV 可达 5-10%，所有未登录词映射到 <unk>，导致信息完全丢失

**太大的影响**:
- **内存爆炸**：$|V| = 200k$，$d_{model} = 4096$，嵌入层参数 $819M$，占 7B 模型总参数的 12%，推理时 GPU 显存占用 3.2GB
- **长尾稀疏**：50% 的词表 token 出现频率 <100 次，嵌入向量训练不充分，"被遗忘的维度"导致质量下降
- **计算浪费**：大词表导致 embedding lookup 成为瓶颈，在 GNN 加速器上 embedding 表读取占推理延迟的 25-30%

**致命弱点**:
- **领域错配**：在法律文本上训练的 32k 词表直接用于医疗，OOV 率从 0.05% 飙升到 2.3%，需领域自适应重训词表
- **静态词表陷阱**：训练完成后词表固定，新增网络词汇（"Covid-19", "LLaMA"）必须用子词组合表示，无法高效吸收新语义

## 5. 工业映射
在工业界，词汇表大小的选择是 **LLM 部署的首个架构决策**：
- **GPT-3**: OpenAI 选择 50k 词表，基于信息熵分析和 A/B 测试：在从 20k 到 100k 的实验中，50k 在训练损失和推理速度（1000 token/s）上达到最优，增大到 100k 仅带来 0.5% 损失下降但推理速度下降 35%
- **LLaMA 2**: Meta 采用 32k 词表，权衡内存占用（embedding 层仅占 1.3B 参数的 5%）和长序列处理（4k 上下文），通过词表量化（INT8）进一步压缩到 128MB，可在边缘设备部署
- **T5 mT5**: Google 多语言词表 250k 是极端案例，覆盖 101 种语言，但通过共享子词（数字、标点）减少冗余，embedding 梯度检查点（gradient checkpointing）优化内存，"太大"的代价由 TPUs 的 HBM 高带宽承担
- **自适应词表**：Claude 使用动态词表扩展，对高频 OOV 词定期（每 1B 训练 token）重训 tokenizer 增量添加 token，平衡静态词表的确定性和动态适应性，在 6 个月迭代中 OOV 率从 0.1% 降至 0.02%
