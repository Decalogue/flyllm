# RAG检索模块设计

## 1. 核心定性

本质上，RAG检索模块是一个为解决LLM知识滞后与幻觉问题，通过**语义对齐+多路召回+重排精筛**三级架构实现的知识外挂系统。

---

## 2. 具体流程

1. **索引构建**：将文档切分为Chunk，经Embedding模型编码为向量，构建**向量索引**（HNSW/IVF）与**倒排索引**（BM25）双路索引体系。
2. **多路召回**：Query并行送入向量检索（语义相似度）与关键词检索（字面匹配），分别召回Top-K候选。
3. **精排融合**：通过RRF（Reciprocal Rank Fusion）或轻量级Cross-Encoder重排模型融合多路结果，输出最终上下文。

---

## 3. 数学基础

**语义相似度（余弦相似度）**：
$$\text{similarity}(q, d) = \frac{\mathbf{e}_q \cdot \mathbf{e}_d}{\|\mathbf{e}_q\| \|\mathbf{e}_d\|}$$

其中：
- $\mathbf{e}_q \in \mathbb{R}^D$：查询向量（Query Embedding）
- $\mathbf{e}_d \in \mathbb{R}^D$：文档向量（Document Embedding）
- $D$：向量维度（通常768或1024）

**RRF融合得分**：
$$\text{RRF}(d) = \sum_{r \in R} \frac{1}{k + \text{rank}_r(d)}$$

其中：
- $R$：召回策略集合（如向量检索、BM25）
- $\text{rank}_r(d)$：文档$d$在策略$r$中的排序位次
- $k$：平滑常数（通常取60）

**HNSW检索复杂度**：
$$T_{\text{search}} = O(\log N), \quad T_{\text{build}} = O(N \cdot \log N \cdot M)$$

其中：
- $N$：向量总数
- $M$：每层最大邻居数（默认16）

## 4. 工程考量

| 维度 | Trade-off | 致命弱点 |
|------|-----------|----------|
| **精度vs速度** | HNSW牺牲建索时间换取对数级查询速度 | 高维稀疏向量（>2048维）下图索引退化，Recall@10暴跌至<60% |
| **语义vs字面** | 向量检索捕获同义改写但丢失精准术语 | 专业缩写/产品型号（如"GPT-4"vs"gpt4"）语义鸿沟导致漏召 |
| **内存vs磁盘** | IVF-PQ量化降低90%内存但引入量化误差 | 极端分布偏移场景（如医疗术语vs通用语料）MRR下降30%+ |
| **实时性** | 增量索引支持实时写入但碎片累积 | 高频更新（>1k doc/s）触发HNSW重建，查询延迟毛刺>500ms |

---

## 5. 工业映射

在工业界，该架构被直接应用于：
- **LangChain/ LlamaIndex** 的 `EnsembleRetriever`：默认组合BM25 + 向量检索，RRF融合
- **OpenAI GPTs / Azure OpenAI**：基于**Qdrant/Pinecone**托管HNSW索引，支持混合搜索（Hybrid Search）
- **字节跳动Coze / 百度千帆**：采用**两阶段检索**（向量召回 + 精排模型），应对百亿级文档的毫秒级检索
- **ElasticSearch 8.x**：原生集成`dense_vector`字段 + `kNN`搜索，支持文本+向量混合查询

### 检索方法选择决策树

```
                        ┌─────────────────┐
                        │   数据规模？     │
                        └────────┬────────┘
                                 │
            ┌────────────────────┼────────────────────┐
           <100万条                            >100万条/实时更新
              │                                      │
              ▼                                      ▼
        ┌─────────────┐                      ┌─────────────┐
        │ 是否需要    │                      │ 用HNSW+增量 │
        │ 精确匹配？  │                      │ 索引(Qdrant)│
        └──────┬──────┘                      └─────────────┘
               │
       ┌───────┴───────┐
      是               否
       │                │
       ▼                ▼
  ┌─────────┐     ┌─────────────┐
  │ BM25    │     │ 向量检索     │
  │ (ES)    │     │ + 重排模型   │
  └─────────┘     └─────────────┘
```

**黄金法则**：
- **冷启动/FAQ类**：纯BM25，低成本高精准
- **语义丰富/开放域**：向量检索（M3E/bge-large）+ Cross-Encoder重排
- **企业级生产**：必上混合检索（Hybrid），召回率提升15-40%
