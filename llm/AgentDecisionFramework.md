# Agent 决策流程与框架选型

## 1. 核心定性
本质上，Agent 决策流程是为了解决"LLM 缺乏自主任务执行与错误恢复能力"的问题，通过**"感知-推理-行动-记忆"闭环控制机制**实现的自主任务处理架构。

## 2. 主流程拆解
1. **感知编码**：将环境输入（用户指令、工具返回、系统状态）编码为上下文表征，触发决策循环
2. **规划推理**：基于目标与记忆状态，选择决策框架（ReAct / Plan-and-Solve / Reflexion）生成动作序列
3. **执行反馈**：调用工具与环境交互，观测结果并更新记忆，迭代直至任务完成或到达终止条件

## 3. 数学基础

### ReAct 框架的形式化定义
$$a_t = \\pi_{\\theta}(o_t, h_{t-1})$$
$$o_{t+1} = \\mathcal{E}(a_t)$$
$$h_t = h_{t-1} \\oplus (a_t, o_{t+1})$$

其中：
- $a_t$: 第 $t$ 步选择的**动作**（Thought / Action / Observation 三元组）
- $o_t$: 第 $t$ 步的**环境观测**（工具返回结果）
- $h_t$: 截止到 $t$ 步的**历史记忆轨迹**
- $\\pi_{\\theta}$: 参数为 $\\theta$ 的 LLM **策略函数**
- $\\mathcal{E}$: **环境执行器**，将动作映射为观测
- $\\oplus$: 记忆更新操作（通常为拼接或摘要压缩）

### 决策框架对比矩阵

| 框架 | 核心机制 | 形式化表达 | 适用场景 |
|------|---------|-----------|---------|
| **CoT** | 链式推理 | $y = f_{\\theta}(x \\oplus \\text{"Let's think step by step"})$ | 单步复杂推理，无需工具 |
| **ReAct** | 推理-行动交错 | $[(T_1, A_1, O_1), ..., (T_n, A_n, O_n)]$ | 多步工具调用，动态规划 |
| **Plan-and-Solve** | 先规划后执行 | $P = \\text{Planner}(g); \\text{Exec}(P)$ | 任务可预先分解，步骤稳定 |
| **Reflexion** | 自我反思纠错 | $a_t = \\pi(o_t, h_{t-1}, r_{t-1})$ | 试错成本高，需要经验积累 |

## 4. 工程考量

### Trade-off 矩阵

| 维度 | 优势 | 牺牲 |
|------|------|------|
| **ReAct vs CoT** | 动态工具调用、错误恢复能力 | Token 消耗增加 30-50%，延迟累积 |
| **Plan-and-Solve vs ReAct** | 一次性生成完整计划，减少 LLM 调用次数 | 计划僵化，无法应对环境突变 |
| **Reflexion vs 基线** | 通过失败经验提升长期成功率 | 需要额外存储失败案例，内存膨胀 |

### 致命弱点

1. **ReAct 的无限循环陷阱**：当工具返回格式异常或策略函数 $\\pi_{\\theta}$ 对 $o_t$ 理解偏差时，Agent 可能在"重试-失败"中死循环，需设置**最大迭代次数**与**退出启发式**

2. **Plan-and-Solve 的级联错误**：规划阶段若出现逻辑漏洞，执行阶段无法自愈，适合**确定性高、步骤可枚举**的任务（如数据 ETL 流水线）

3. **上下文窗口爆炸**：$h_t$ 随交互线性增长，超过 LLM 上下文限制时需引入**记忆压缩**（摘要、向量检索）或**分层架构**（Manager-Worker Agent 树）

## 5. 工业映射

在工业界，该机制被直接应用于：

| 项目 | 框架映射 | 应用场景 |
|------|---------|---------|
| **LangChain Agent** | ReAct + Tool 抽象 | 企业 RAG 系统、多 API 编排 |
| **AutoGPT** | ReAct + 长期记忆（向量库） | 自主研究任务、代码生成 |
| **BabyAGI** | Plan-and-Solve + 任务队列 | 持续运行的后台自动化任务 |
| **Voyager (OpenAI)** | Reflexion + 技能库 | Minecraft 自主探索、代码技能习得 |
| **GPT-4o 函数调用** | CoT + 确定性路由 | 单次工具调用，低延迟场景 |

---

### 选型决策树

```
任务是否需要工具调用？
├── 否 → 使用 CoT / ToT（纯推理）
└── 是 → 步骤是否可预先确定？
    ├── 是 → Plan-and-Solve（减少 LLM 调用）
    └── 否 → 是否需要从失败中学习？
        ├── 是 → Reflexion（试错迭代）
        └── 否 → ReAct（通用默认选择）
```

> **Senior 建议**：生产环境首选 **ReAct + 退出守卫**（最大步数/时间限制），在验证集上测量成功率-Token 消耗曲线，找到成本与效果的帕累托前沿。
