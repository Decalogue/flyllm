# RAG（检索增强生成）

## 1. 核心定性

本质上，**RAG** 是一个为了解决**大模型知识滞后与幻觉**问题，通过**动态检索外部知识库**来**扩展上下文并约束生成**的架构范式。

## 2. 具体流程

1. **索引阶段**：将外部文档切分为 Chunk，通过 Embedding 模型编码为向量，存入向量数据库构建可检索的知识索引。
2. **检索阶段**：用户查询经相同编码器转为查询向量，在向量空间中进行相似度搜索，召回 Top-k 相关文档片段。
3. **增强生成**：将检索到的上下文与原始查询拼接，注入 Prompt，引导模型基于检索证据生成答案。

## 3. 数学基础

**核心检索目标**（向量相似度）：
$$\text{Retrieval}(q, D) = \underset{d \in D}{\text{top-k}} \ \text{sim}(E_q(q), E_d(d))$$

**生成概率建模**：
$$P(y|q) = \sum_{d \in \mathcal{R}(q)} P(d|q) \cdot P(y|q, d)$$

其中：
- $E_q, E_d$: 查询与文档的编码器（通常共享参数）
- $\text{sim}(\cdot, \cdot)$: 余弦相似度或点积相似度函数
- $\mathcal{R}(q)$: 针对查询 $q$ 召回的文档集合
- $P(d|q)$: 检索器对文档 $d$ 的相关性评分（可视为注意力权重）
- $P(y|q, d)$: 基于查询与检索上下文生成答案的条件概率

## 4. 工程考量

| 维度 | Trade-off | 致命弱点 |
|------|-----------|----------|
| **时效性** | 牺牲静态知识容量换取动态更新能力 | 检索质量决定生成质量 —— 检索失败时幻觉反而加剧 |
| **上下文窗口** | 召回更多文档提升覆盖，但超出 LLM 上下文限制 | **检索噪声**：无关文档引入会严重污染生成结果 |
| **延迟** | 增加检索轮次换取答案准确性 | 端到端延迟 = 向量检索 RTT + LLM 生成时间，高并发下成为瓶颈 |
| **一致性** | 外部知识更新导致同一问题答案漂移 | **知识冲突**：检索到矛盾信息时模型可能"和稀泥" |

## 5. 工业映射

在工业界，该机制被直接应用于：
- **LangChain / LlamaIndex**：标准化 RAG Pipeline 编排框架
- **OpenAI GPTs / 文心一言插件**：实现知识库问答与私有数据接入
- **Perplexity.ai / 必应 Copilot**：实时检索互联网信息消除知识截止日期限制
