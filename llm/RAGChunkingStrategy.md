# RAG 文档分块策略设计

## 1. 核心定性

**本质上，文档分块是一个为了在 Embedding 模型的上下文限制与语义完整性之间取得平衡，通过控制文本粒度来最大化检索召回率与答案相关性的工程策略。**

---

## 2. 具体流程

1. **文本解析与清洗**：将原始文档（PDF/Word/HTML）解析为纯文本，去除页眉页脚、表格噪点，保留段落结构。
2. **分块边界切割**：按固定长度、语义段落或递归层级进行切割，决定是否保留块间重叠（Overlap）以维护上下文连续性。
3. **元数据注入与索引**：为每个 Chunk 附加源文档ID、页码、标题等元数据，经 Embedding 模型向量化后写入向量数据库。

---

## 3. 分块方法对比（数学刻画）

| 方法 | 核心公式/逻辑 | 适用场景 |
|------|--------------|---------|
| **固定字符数** | $\text{Chunk}_i = S[i \cdot L : (i+1) \cdot L + O]$ | 通用型，O 为 Overlap 长度 |
| **递归字符** | $\text{优先按}\n\n\text{分割，其次按}\n\text{，最后按空格}$ | 结构化文档（Markdown/代码）|
| **语义段落** | $\text{Chunk} = \{sentences \mid SemanticScore(s_i, s_{i+1}) > \theta\}$ | 长文本，需保持语义完整 |
| **Agentic 分块** | $\text{LLM}(Doc) \rightarrow [Chunk_1, Chunk_2, ...]$ | 极高精度要求，成本不敏感 |

**Overlap 计算**（维护上下文连续性）：
$$\text{Overlap Ratio} = \frac{O}{L} \times 100\%$$
其中：
- $L$: 目标块长度（token/字符）
- $O$: 重叠长度，通常取 $0.1L \sim 0.3L$

---

## 4. 块大小选择：Trade-off 分析

### 关键公式：有效信息密度
$$\text{Efficiency} = \frac{\text{RelevantTokens}}{\text{TotalTokens}} = f(L, \text{EmbeddingModel})$$

| 块大小 | 优点 | 致命弱点 |
|--------|------|---------|
| **小（<256 tokens）** | 检索精度高，噪音少 | 语义割裂，丢失跨句依赖 |
| **中（512~1024 tokens）** | 平衡精度与召回 | 需调优 Overlap，存在边界截断风险 |
| **大（>2048 tokens）** | 完整保留上下文 | 检索噪声高，Embedding 稀释效应严重 |

**块大小选择决策树**：
```
Embedding 模型上下文 → 1024 tokens（text-embedding-ada-002）
           ↓
    目标块大小 ≤ 512 tokens（保留 50% 余量）
           ↓
    是否需要跨段落推理？ → 是：Overlap = 20%；否：Overlap = 0
           ↓
    文档类型：代码 → 按函数/类；论文 → 按章节；对话 → 按轮次
```

---

## 5. 工程考量与致命弱点

### Trade-off 矩阵
- **精度 vs 召回**：小块精度高但可能漏检跨块信息；大块召回全但引入噪声。
- **存储 vs 延迟**：块越小，Chunk 数量越多，向量库存储压力↑，检索延迟↑。
- **成本 vs 质量**：Agentic 分块效果最好，但调用 LLM 的成本是固定分块的 100x+。

### 致命弱点（边界熔断场景）
| 场景 | 退化表现 |
|------|---------|
| **表格/代码跨页** | 固定字符切割导致语法断裂，语义完全丢失 |
| **指代消解** | "上述方法"、"前文提到"在小块中成为死链 |
| **多语言混合** | 不同语言的 token 密度差异大，固定长度导致分块不均 |
| **超长文档（书籍）** | 扁平索引无法表达层级结构，检索陷入局部最优 |

---

## 6. 工业映射

- **LangChain**: `RecursiveCharacterTextSplitter` 实现递归层级分块，支持自定义分隔符优先级。
- **LlamaIndex**: `SentenceWindowNodeParser` 实现语义窗口分块，自动维护前后句关系。
- **OpenAI Embedding**: 官方推荐 512~1024 tokens/块，配合 `text-embedding-3-small` 的 8k 上下文。
- **RAPTOR (Stanford)**: 递归摘要 + 树形索引，解决长文档层级检索问题。
