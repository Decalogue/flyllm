# Tool Calling：模型如何学会使用工具

---

## 1. 核心定性

**本质上，Tool Calling 是一个为了让 LLM 突破参数化知识边界，通过 "意图识别 → 参数填充 → 外部执行 → 结果回注" 闭环机制实现的函数调用编排系统。**

---

## 2. 具体流程

1. **工具注册**：将外部工具（API/函数）以结构化 Schema 注册到系统，包含名称、描述、参数类型及约束
2. **意图触发**：用户 Query 经模型编码后，与工具描述做语义匹配，输出 `<tool_call>` 标签及 JSON 参数
3. **执行回注**：系统解析标签，调用真实工具，将返回结果拼接回上下文，触发模型生成最终回答

---

## 3. 数学基础

**决策公式（工具选择概率）：**

$$P(t_i|q) = \frac{\exp(\text{sim}(E_q, E_{t_i}) / \tau)}{\sum_{j=1}^{N} \exp(\text{sim}(E_q, E_{t_j}) / \tau)}$$

**其中：**
- $q$: 用户查询的语义嵌入
- $t_i$: 第 $i$ 个工具的嵌入向量（由描述文本编码得到）
- $\text{sim}(\cdot)$: 余弦相似度函数
- $\tau$: 温度系数（控制选择锐度）
- $N$: 工具库中候选工具总数

**参数生成（自回归解码）：**

$$\theta^* = \arg\max_{\theta} \sum_{(q,a,t) \in \mathcal{D}} \log P_{\theta}(a, t | q, \mathcal{T})$$

- $\theta$: 模型参数
- $\mathcal{T}$: 所有工具描述构成的上下文
- $(q,a,t)$: 查询-答案-工具调用三元组训练数据

---

## 4. 工程考量

| Trade-off | 牺牲 | 换取 |
|-----------|------|------|
| **Schema 严格性** | 灵活性（难以处理模糊需求） | 可解析性（100% 结构化输出） |
| **工具数量** | 单次推理的精度（过多工具导致注意力稀释） | 系统能力覆盖广度 |

**致命弱点：**
- **幻觉调用**：模型可能为不存在的需求虚构工具调用（幻觉 JSON）
- **参数污染**：长上下文下，工具返回的大段结果可能淹没用户原始意图
- **级联错误**：上游工具输出错误会被模型当作事实继续传播，缺乏自动熔断机制

---

## 5. 工业映射

在工业界，该机制被直接应用于：
- **OpenAI GPT-4 Function Calling / Claude Tool Use**：原生支持函数调用标签注入与解析
- **LangChain / LlamaIndex**：通过 `BaseTool` 抽象层实现工具注册、路由、重试的编排框架
- **AutoGPT / BabyAGI**：多工具链自主规划，应对复杂任务分解与长期记忆场景

---

**一句话总结**：Tool Calling = 用 **语义相似度** 做工具路由，用 **受限解码** 做参数生成，用 **执行器隔离** 保安全边界。
