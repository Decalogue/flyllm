# 局部注意力适用于什么场景？窗口大小怎么设计？

## 1. 核心定性
本质上，**局部注意力**是一种基于局部性假设的稀疏注意力机制，通过将每个 token 的注意力范围限制在固定大小的滑动窗口内，将复杂度从 $O(n^2)$ 降至 $O(n w)$，在捕获短距离依赖的同时，依赖层级堆叠实现远距离信息传播。

## 2. 具体流程
1. **窗口设计**：为每个位置 $i$ 定义局部窗口 $[i-w, i+w]$，仅计算该窗口内 token 的注意力权重，对当前 token 的固定大小邻域进行密集交互
2. **层级传播**：在深层 Transformer 中，底层捕获局部模式（如短语结构），中层聚合区域信息（如句子成分），顶层通过多跳传播获取全局上下文，理论证明 12 层可捕获距离 $12w$ 的依赖
3. **边界处理**：序列首尾采用对称填充或截断窗口，确保窗口大小恒定；同时插入锚点 token（如 [CLS]）作为全局信息汇，打破纯局部限制

## 3. 数学基础
**滑动窗口掩码**:
$$M_{ij} = \begin{cases}
1 \iff |i - j| \leq w 
\end{cases}$$

**复杂度**:
$$\text{Time: } O(n w d), \quad \text{Memory: } O(n w)$$

**窗口大小选择**:
经验公式：
$$w = \alpha \cdot \sqrt{d} \approx 64 \text{ to } 256 \text{ for } d=1024$$

其中任务相关：
- 文本分类/NER: $w = 32-64$
- 机器翻译: $w = 128-256$
- 文档 QA: $w = 256-512$

## 4. 工程考量
**适用场景**:
- **长文本理解**：文档分类、长序列标注，内存从 16GB 降至 2GB
- **局部性强的任务**：NER、词性标注、句法分析，90% 依赖在窗口内
- **实时推理**：流式生成中，局部注意力天然适配增量解码

**窗口大小设计原则**:
- **经验法则**：$w = 128$ 是通用起点
- **任务适配**：在验证集上尝试 $w \in \{32, 64, 128, 256, 512\}$
- **层级补偿**：深层网络可用小窗口 $w=64$，浅层网络需大窗口 $w=256$

**Trade-off**:
- **局部性 vs 全局性**：窗口越小，计算越快，但远距离依赖捕获能力越差
- **内存 vs 质量**：$w=512$ 内存占用是 $w=128$ 的 4 倍，但 GLUE 得分仅提升 1.2%

**致命弱点**:
- **长尾依赖失效**：无法显式捕获超窗口依赖
- **窗口边界伪影**：边界 5% token 的 perplexity 上升 10-15%
- **层级传播延迟**：信息经过 L 次变换，信噪比下降

## 5. 工业映射
**Longformer 在 AllenAI**：处理 4096 token 科学文献，$w=256$，在 TriviaQA 上 F1 67.2（vs BERT 63.1），内存 4.2GB→1.1GB

**StreamingLLM**：推理时保持 4k token 滑动窗口，配合 attention sink，实现无限长度生成

**Huawei Pangu**：层次化窗口，底层 $w=64$，中层 $w=256$，顶层 $w=512$，在中文文档分类上准确率 94.7%

**自适应窗口**：动态调整窗口大小，特殊 token 用大窗口 $w=512$，普通 token 用小窗口 $w=128$，效果提升 2-3%
