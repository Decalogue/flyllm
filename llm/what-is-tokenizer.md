# 什么是分词器？为什么 LLM 需要分词而不是直接处理字符？

## 1. 核心定性
本质上，**分词器（Tokenizer）** 是将文本序列离散化为紧凑语义单元（token）的确定性映射函数，通过平衡词汇表大小与序列长度来缓解高维字符空间的维度灾难问题。

## 2. 具体流程
1. **训练阶段**：基于语料统计信息（如 BPE 的字符频数、Unigram 的句概率），通过合并或剪枝算法从字符级逐步构建词汇表
2. **编码阶段**：对输入文本应用最长匹配原则，将字符串拆分为词汇表中存在的 token 序列，未登录词回退到子字符或字节级
3. **嵌入查表**：将离散的 token ID 序列映射为密集嵌入向量输入 Transformer 的嵌入层

## 3. 数学基础
设字符集 $\Sigma$ 上的原始文本为 $S = c_1c_2...c_n$，分词器定义映射函数：
$$\text{Tokenize}(S) = [t_1, t_2, ..., t_m] \text{ where } m \leq n$$

- **输入维度灾难**：若直接处理字符，序列长度为 $n$，嵌入矩阵为 $|\Sigma| \times d_{model}$。对于 Unicode（$|\Sigma| \approx 150k$），显存占用极高
- **token 压缩率**：压缩比 $r = n/m$，典型值在 3-5 之间。词汇表大小 $|V|$ 满足信息论约束：
$$H(S) \approx \sum_{t \in V} -p(t)\log p(t) \leq \log|V|$$

## 4. 工程考量
**Trade-off**:
- **压缩 vs 信息丢失**：词汇表过小导致频繁分词成罕见子词，增加序列长度；过大则稀有词嵌入训练不充分（长尾分布问题）
- **碎片化成本**：分词后序列长度增加，Transformer 的自注意力计算复杂度为 $O(m^2d)$，直接和平方级增长

**致命弱点**:
- **语言失衡**：像中文这样的无空格语言，分词错误或过度切分会导致语义漂移，例如"人工智能"被切分为"人+工+智能"
- **推理歧义**：下游任务对 token 边界敏感，如"New York"应保持完整还是拆开直接影响实体识别

## 5. 工业映射
在工业界，分词器是**所有 LLM 推理服务的首层瓶颈**：
- **Hugging Face Tokenizers**: Rust 实现的重构库，通过并行编码和预计算将速度提升 10-100 倍，支持训练后保存为 JSON 格式供部署使用
- **OpenAI GPT-4**: 使用 BPE 变体（tiktoken），词汇表约 100k，token 编码规则直接暴露在 tiktoken 库中，实现闭源模型的透明化调用
- **边界处理**: Byte-level BPE（BBPE）通过 256 字节词汇表回退机制，确保任何 UTF-8 文本都可编码，解决多语言支持难题
