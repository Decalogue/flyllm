# RAG中QueryExpansion查询扩展与重写机制

## 1. 核心定性

本质上，**Query Expansion** 是为了解决**用户原始查询语义稀疏/偏移**导致的检索失效问题，通过**生成语义等价或互补的查询变体**来扩展检索覆盖面的**查询增强策略**。

## 2. 具体流程

1. **Query Rewriting（查询重写）**：利用 LLM 或规则引擎将原始查询 $q$ 改写为语义等价但表达方式不同的多个查询变体 $\{q_1, q_2, ..., q_n\}$，消除歧义并补充隐含意图；
2. **Multi-Query Retrieval（多路检索）**：并行执行各变体查询，分别从向量数据库中检索 Top-K 文档片段；
3. **Fusion & Rerank（融合重排）**：基于 RRF (Reciprocal Rank Fusion) 或多路召回分数加权，对合并结果去重重排后输入生成器。

## 3. 数学基础

**RRF 融合公式**（多查询结果聚合）：

$$\text{RRF}(d) = \sum_{q_i \in Q} \frac{1}{k + r_i(d)}$$

其中：
- $Q$: 扩展后的查询集合 $\{q_0, q_1, ..., q_m\}$（含原始查询）
- $r_i(d)$: 文档 $d$ 在查询 $q_i$ 结果列表中的排名（从1开始）
- $k$: 平滑常数（通常取60）

**查询重写生成的条件概率**：

$$q^* = \arg\max_{q'} P(q' | q, \mathcal{I}) \approx \text{LLM}_{\text{rewrite}}(q, \text{prompt}_{\text{expansion}})$$

其中 $\mathcal{I}$ 代表任务指令（如"生成3个语义等价但用词不同的查询"）。

## 4. 工程考量

| Trade-off | 说明 |
|-----------|------|
| **召回率↑ vs 精确率↓** | 扩展查询增加了检索覆盖面，但引入噪声文档的概率同步上升，需配合重排模型（Cross-Encoder）过滤 |
| **延迟 vs 覆盖度** | 每增加一个查询变体，向量检索延迟线性增长 $O(n \cdot T_{\text{search}})$，需控制扩展数量（通常 3-5 个） |
| **重写质量依赖** | LLM 生成的扩展查询可能与原意偏离（幻觉查询），需设置语义相似度阈值过滤：$\text{sim}(q, q') > \theta$ |

**致命弱点**：
- **长尾查询失效**：当原始查询本身极度模糊或领域特定时，LLM 重写的查询可能集体偏离用户真实意图，导致"一致但错误"的检索结果；
- **高并发成本爆炸**：每个请求触发 N 次向量检索 + 1 次 LLM 重写，QPS 承载能力同比下降。

## 5. 工业映射

在工业界，该机制被直接应用于：
- **LangChain MultiQueryRetriever**：自动将用户查询扩展为 3-5 个变体并行检索，应对模糊查询；
- **RAG-Fusion 架构**：结合 RRF 融合多路召回结果，显著优于单路向量检索的基线；
- **百度/阿里搜索的 Query 理解层**：在向量检索前通过 Seq2Seq 模型生成同义 query，提升垂类搜索召回率 15-30%。
