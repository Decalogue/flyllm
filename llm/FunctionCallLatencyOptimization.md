# Function Call 延迟优化与准确性权衡

---

## 1. 核心定性

本质上，Function Call 是一个**解码时工具选择 + 结构化输出**的联合优化问题，延迟瓶颈在于工具空间大时的 token-by-token 解码成本，准确性瓶颈在于工具描述的语义对齐与边界分类。

---

## 2. 具体流程

1. **工具检索**：用户 Query → Embedding 检索 Top-K 候选工具，将候选空间从 $N$ 压缩到 $k \ll N$；
2. **模式注入**：将候选工具的 JSON Schema 注入 System Prompt，触发模型生成带 `<tool_call>` 标记的结构化输出；
3. **并行执行**：解析工具调用 → 并发执行 → 结果回注 → 二次解码生成最终回答。

---

## 3. 数学基础

**延迟构成公式**：

$$T_{total} = T_{TTFT} + \sum_{i=1}^{L} T_{token}^{(i)} + T_{tool} + T_{RT}$$

其中：
- $T_{TTFT}$: Time To First Token，首 token 延迟
- $L$: 输出 token 长度
- $T_{token}^{(i)}$: 第 $i$ 个 token 的解码耗时（与 KV-Cache 长度线性相关）
- $T_{tool}$: 工具执行时间（可并行化）
- $T_{RT}$: 往返网络延迟

**工具检索的准确率-召回率权衡**：

$$F_\beta = (1+\beta^2) \cdot \frac{Precision \cdot Recall}{\beta^2 \cdot Precision + Recall}$$

- $\beta > 1$: 偏重召回（宁可多召回也要避免漏工具）
- $\beta < 1$: 偏重精确（减少无关工具带来的噪音）

---

## 4. 工程考量

### Trade-offs

| 优化手段 | 速度收益 | 准确性代价 | 适用场景 |
|---------|---------|-----------|---------|
| **工具检索剪枝** | $O(N) \to O(k)$ | 可能漏选边缘工具 | 工具数 > 50 |
| **小模型路由** | 降低 10-50x 延迟 | 分类错误率上升 | 高频/简单意图 |
| **Schema 蒸馏** | 缩短 prompt | 丢失字段细节约束 | 字段数 > 20 |
| **投机解码** | 2-3x 加速 | 工具参数幻觉风险 | 固定格式调用 |

### 致命弱点

1. **工具冲突地狱**：当多个工具功能重叠（如 `search_weather` vs `get_forecast`），小模型路由极易误判，导致"调对了工具但给了错参数"的**静默失败**；
2. **长 Schema 崩溃**：当单工具字段 > 50 个，KV-Cache 膨胀导致 $T_{token}$ 指数级上升，首 token 延迟可能突破 2s；
3. **并发工具爆炸**：用户意图需调用 5+ 工具时，串行执行 $T_{tool}$ 累积，并行执行又可能触发下游限流。

---

## 5. 工业映射

- **OpenAI / Claude**: 采用原生 Function Calling 模式，模型内部已做工具空间压缩，延迟约 500-1500ms，适合 10-20 个工具场景；
- **LangChain/LlamaIndex**: 引入 `AgentExecutor` 的 **ReAct 循环**，牺牲速度换取多步推理能力，单轮延迟 2-5s；
- **阿里 Qwen / 百度文心**: 采用 **意图识别 → 小模型路由 → 大模型执行** 的三层架构，工具数扩展到 100+ 仍保持 < 800ms 首 token；
- **向量检索层**: 使用 Milvus / Pinecone 做工具 Embedding 召回，召回阶段延迟控制在 10-50ms，是大规模工具集的**标配基础设施**。
