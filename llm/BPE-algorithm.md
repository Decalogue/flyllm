# BPE 算法的核心思想是什么？能详细说说它的训练和编码过程吗？

## 1. 核心定性
本质上，**BPE（Byte Pair Encoding）** 是一种基于贪婪合并策略的无监督子词切分算法，通过迭代合并最高频的字符对来构建变长词汇表，在压缩文本长度与引入语义边界之间取得平衡。

## 2. 具体流程
1. **训练阶段**：从字符级词汇表开始，统计所有相邻字符对在语料中的共现频率，每轮合并最高频的字符对并更新词汇表，直到达到目标词汇表大小或合并次数
2. **合并优先级**：使用优先级队列维护候选对频率，每次合并后仅需重新计算受影响窗口的共现计数，时间复杂度为 $O(V \cdot L)$ 其中 $V$ 为目标词表大小，$L$ 为平均序列长度
3. **编码阶段**：对待编码文本重复执行最长匹配原则，查找当前词汇表中存在的最长子串进行切分，未匹配字符回退到字节级表示

## 3. 数学基础
设字符集 $C = \{c_1, c_2, ..., c_n\}$，初始语料 $S$ 中每个字符作为独立 token：
$$V^{(0)} = C, \quad \text{tokenize}^{(0)}(S) = \text{characters}(S)$$

在训练第 $t$ 步，统计相邻 token 对频率：
$$F^{(t)}(x,y) = \sum_{i=1}^{m-1} \delta(\text{token}_i^{(t)}=x \land \text{token}_{i+1}^{(t)}=y)$$

选择最高频对合并：
$$(x^*, y^*) = \arg\max_{(x,y)} F^{(t)}(x,y)$$
$$V^{(t+1)} = V^{(t)} \cup \{x^*y^*\}$$

合并后更新 token 序列：
$$\text{tokenize}^{(t+1)}(S) = \text{tokenize}^{(t)}(S) \text{ with all } x^*y^* \text{ merged}$$

**编码复杂度**:
- 训练：$O(N \cdot T)$ 其中 $N$ 为语料字符数，$T$ 为合并轮数
- 推理：$O(L \cdot |V|)$ 使用 Trie 树或哈希表加速最长匹配

## 4. 工程考量
**Trade-off**:
- **频率 vs 语义**：仅基于统计频率可能合并非语义单元（如"ing"），在英语中有效但在黏着语（土耳其语、芬兰语）中效果差
- **词汇表膨胀**：高频词汇（如"the"）可能形成超长 token，挤压低频词空间，导致长尾词被过度切分

**致命弱点**:
- **冷启动问题**：新词出现时必须拆分为已知子词，无法保留完整语义。例如专有名词"Baichuan"会被切分为"Baich+uan"，破坏完整性
- **顺序依赖**：训练时的合并顺序直接影响最终分词结果，不同语料训练的 BPE 词表无法直接兼容，导致跨模型迁移困难

## 5. 工业映射
在工业界，BPE 是**开源 LLM 分词的事实标准**：
- **GPT 系列**：OpenAI 使用 BPE 训练 tiktoken，词表大小 ~100k，通过预计算合并规则实现微秒级编码。训练从 4B token 语料开始迭代合并，每轮采样高频对减少计算量
- **LLaMA 实现**：SentencePiece 库中的 BPE 变体，引入字节级回退（Byte-level BPE），对任何 UTF-8 文本保证 100% 覆盖率。训练时使用前缀树（prefix tree）将复杂度优化到 $O(N)$
- **Hugging Face AutoTokenizer**：tokenizer.json 中存储合并规则和词汇表，加载后使用 Rust 实现的快速分词器并行处理批次文本，在 8 核 CPU 上可达 10k+ doc/s
